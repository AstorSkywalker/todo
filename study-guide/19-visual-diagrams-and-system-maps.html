Lesson 19

Visual Diagrams and System Maps

This lesson turns the project into richer visual maps so you can study the application as a system, not only as a collection of files. The goal is to help your eyes understand the architecture before your brain has to parse every line of code.

Lesson Contents

  • Request flow map
  • Frontend render map
  • Storage sync map
  • localStorage map
  • Future Oracle path
  • Exercise and checkpoint

1. Request Flow Map

Request Lifecycle

Browsersubmit form or fetch data
→
server.jsparse URL and dispatch
→
routes/todos.jsmatch route and method
→
todoServicerun business logic

Read this as one story: the browser asks, the route layer decides, and the service layer does the real work.

This diagram is intentionally simple, but it already removes a lot of confusion. It shows that requests do not jump directly from the browser into storage. They pass through clear layers first.

2. Frontend Render Map

Render Lifecycle

initialize()restore preferences and bind events
loadTodos()fetch current data from the API
renderDashboard()counters, chart, groups, tasks
DOM updatescards, filters, compact table, chart labels

This flow helps you separate startup, data loading, rendering, and visible output into distinct phases.

The frontend is not random DOM manipulation. It follows a pipeline: restore state, load data, then paint the screen from that state.

3. Storage Sync Map

Seeing the sync flow like this makes the architecture easier to remember: CSV is still trusted first, and SQLite follows it.

6. Exercise and Checkpoint

Problem

Draw your own version of the request flow for POST /api/todos on paper or in a notes app.

Solution Approach

Your diagram should show at least these steps in order:

Browser form
→ server.js
→ routes/todos.js
→ todoService.createTodo()
→ csvService.writeTodos()
→ sqliteService.syncTodosToSqlite()
→ JSON response
→ frontend re-render

How It Should Look

If your map has a clear beginning, middle, and end, and shows where validation and storage happen, then you are thinking about the project as a system instead of isolated files.

Checkpoint: if you can draw the system from memory, you understand it at a much deeper level than someone who only reads individual files.
