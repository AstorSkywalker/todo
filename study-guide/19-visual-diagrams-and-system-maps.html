Lesson 19

Visual Diagrams and System Maps

This lesson converts the project into visual system maps so you can study it with your eyes first, then with code second. That often makes complex projects much easier to own.

Lesson Contents

1. Request Flow Map

→
server.jsdispatch request

This is the basic request path. Once you see it visually, the backend becomes less mysterious.

2. Frontend Render Map

initialize()restore preferences and bind events
loadTodos()fetch current data from the API
renderDashboard()counters, chart, groups, tasks
DOM updatescards, filters, compact table, chart labels

The frontend is driven by a simple idea: restore state, load data, render visible output.

3. Storage Sync Map

Form / API writecreate, update, delete, import
→
CSV writeprimary source of truth
→
SQLite syncmirror current rows

This is one of the project’s most important architectural stories: CSV remains primary while SQLite mirrors it for future evolution.

6. Exercise and Checkpoint

Problem

Draw your own version of the request flow for POST /api/todos on paper or in a notes app.

Solution Approach

Your diagram should show at least these steps in order:

Browser form
→ server.js
→ routes/todos.js
→ todoService.createTodo()
→ csvService.writeTodos()
→ sqliteService.syncTodosToSqlite()
→ JSON response
→ frontend re-render

How It Should Look

If your map has a clear beginning, middle, and end, and shows where validation and storage happen, then you are thinking about the project as a system instead of isolated files.

Checkpoint: if you can draw the system from memory, you understand it at a much deeper level than someone who only reads individual files.
