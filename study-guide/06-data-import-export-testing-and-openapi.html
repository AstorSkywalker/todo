Lesson 6

Data, Import/Export, Testing, and OpenAPI

This lesson covers the data layer and quality layer: CSV parsing, SQLite synchronization, import/export flows, automated tests, OpenAPI structure, and the themed Swagger UI page.

1. CSV Parsing and Escaping

CSV looks simple, but correct handling matters. A task description may contain commas or quotes. That means you cannot just split each line on every comma.

The project includes custom logic to:

  • read quoted CSV values correctly
  • unescape doubled quotes
  • re-escape values when exporting or writing back

This is a strong teaching example because it shows why ï¿½simple text formatsï¿½ still need careful implementation.

2. SQLite As A Mirror

SQLite is not the primary source of truth yet. It is a synchronized mirror.

This teaches a useful transitional architecture:

  • do not break the current app
  • add a second storage layer safely
  • keep future migration options open

The synchronization strategy is straightforward:

  1. read all CSV tasks
  2. replace the SQLite table contents
  3. update the last sync timestamp

3. Import and Export Flows

Import/export teaches how to think about data boundaries.

Export endpoints return either:

  • CSV content
  • JSON content

Import endpoints accept:

  • a format
  • a mode (append or replace)
  • the file content as a string

The import process normalizes rows, validates them, generates unique ids if necessary, and writes the final merged dataset.

4. Automated Tests

The automated test runner is intentionally lightweight. Instead of adding a large test framework, the project uses a custom Node script with assertions.

This is educational because it shows that good testing is more about disciplined scenarios than about tool complexity.

The tests currently cover:

  • CSV parsing edge cases
  • validation rules
  • CRUD endpoint flow
  • grouping endpoint behavior
  • filtering behavior
  • negative API cases
  • import/export round trips
  • storage sync totals

5. OpenAPI Specification

openapi.json describes the API in a machine-readable way. It includes:

  • metadata like title and version
  • server URL
  • paths and operations
  • parameters
  • request bodies
  • response schemas
  • shared components

This is one of the most professional parts of the project because it turns a working API into a documented contract.

6. Swagger UI

docs.html renders Swagger UI using the local OpenAPI file. The project also improves that page by:

  • adding the same light/dark mode behavior
  • adding the same color scheme selector
  • storing those choices in the same browser keys as the main app

This is a very nice lesson about consistency. Documentation should feel like part of the product, not like a separate ugly tool.

7. Why This Quality Layer Matters

A lot of beginner projects stop once the interface works. This project goes further by teaching quality disciplines:

  • test the critical behaviors
  • document the API
  • show storage status
  • make architecture maintainable

These are the habits that move a project closer to professional practice.
