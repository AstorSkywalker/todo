Lesson 24

Debugging Workflows

Learn how to debug the app calmly and methodically by moving between browser tools, network inspection, backend logs, tests, and storage checks.

1. Debugging Mindset

Good debugging is not random guessing. It is a process of narrowing down where a problem starts.

For this project, a useful first question is:

  • is the problem visual only?
  • is the browser state wrong?
  • did the request fail?
  • did the backend reject or transform the data?
  • did storage save the wrong thing?

Core principle: trace one feature from UI to network to backend to storage instead of changing many things at once.

2. Browser-Side Debugging

When the UI looks wrong, start in the browser.

  • Elements panel: inspect markup, classes, and computed styles.
  • Console: inspect JavaScript errors and quick variable output.
  • Application panel: inspect localStorage values like theme, palette, sort mode, or collapsed sections.
// simple debugging pattern
console.log('current filters', appState.filters);
console.log('current view mode', appState.viewMode);

If a feature seems visually broken, check whether the DOM class changed as expected. Sometimes the logic is correct and only the CSS is wrong.

3. Network and API Debugging

If the UI is not updating, the next place to check is the browser network tab.

For example, if deleting a task fails, inspect:

  • which request was sent
  • the request URL
  • the HTTP method
  • the response status
  • the JSON response body

You can also debug outside the browser with quick HTTP checks:

Invoke-RestMethod http://localhost:3000/api/todos/storage
Invoke-RestMethod http://localhost:3000/openapi.json

That helps separate frontend issues from backend issues.

4. Backend Tracing

If the request reaches the server but the result is wrong, the next step is to follow the backend flow:

  1. server.js receives the request
  2. routes/todos.js matches the route
  3. services/todoService.js performs the business logic
  4. services/csvService.js and sqliteService.js update storage

A simple debug habit is to log one value at each layer:

console.log('route payload', payload);
console.log('validated payload', validatedPayload);
console.log('saved todo id', createdTodo.id);

This lets you see whether a bug is introduced during route parsing, validation, service logic, or storage writes.

5. Storage Debugging

Because this app uses both CSV and SQLite, storage debugging has one extra step: check whether both layers still agree.

The project already exposes a helpful route for this:

GET /api/todos/storage

If counts disagree, the next question becomes whether the CSV write failed, the SQLite mirror failed, or the startup sync did not run as expected.

If the result looks wrong, check each stage one at a time instead of guessing.

6. A Reliable Debugging Workflow

A calm, repeatable workflow for this project looks like this:

  1. reproduce the problem on purpose
  2. decide whether it looks visual, client-side, network-related, backend-related, or storage-related
  3. inspect the browser console and network tab
  4. trace the backend route and service path
  5. run the automated tests if the bug may affect shared logic
  6. verify the fix and repeat the same action once more

That is much safer than changing many files at once and hoping the bug disappears.

7. Exercise and Checkpoint

Problem

A user clicks a filter pill and nothing changes in the task list. Describe the debugging path you would follow.

Solution Approach

A good answer should trace the feature from browser interaction to rendered output.

Debugging path:

1. Reproduce the bug and watch the browser console for errors.
2. Inspect the clicked element to confirm the active class changes.
3. Log the related filter state in app.js.
4. Confirm the render function runs after the state update.
5. Check whether the filtered task array changes.
6. If it does change, inspect CSS or DOM rendering.
7. If it does not change, inspect the filter logic itself.

How It Should Look

Your answer should read like a trace plan, not a guess. A strong debugging answer narrows the bug to one layer at a time.

Checkpoint: if you can trace a UI bug from click event to state change to render output to storage or backend when needed, you understand the debugging discipline behind this project.
