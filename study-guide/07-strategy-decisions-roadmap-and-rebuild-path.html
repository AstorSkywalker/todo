Lesson 7

Strategy, Design Decisions, Roadmap, and Rebuild Path

This final lesson explains the strategic thinking behind the project, why certain choices were made in a particular order, and how to rebuild the whole app as a learning exercise from the first line to the final polish.

1. Key Design Decisions

The project made a series of strategic choices. Each one had a reason.

DecisionWhy it was chosenWhat it teaches
Vanilla frontendKeeps concepts visible and direct.You can build rich UIs without hiding logic behind a framework.
CSV-first persistenceFast to understand and inspect.Start with transparency and simplicity.
SQLite mirrorAdds a more realistic data layer without abandoning the original app.Migrations can be incremental.
Modular backend laterAvoid premature complexity.Refactor when growth justifies it.
OpenAPI after stabilizationDocumentation is easier when endpoints are stable.Sequence matters.

2. Tradeoffs and Honest Limitations

No project design is perfect. Good engineering means understanding tradeoffs.

  • CSV is easy, but not ideal for concurrency.
  • Vanilla JavaScript is transparent, but becomes heavier as the UI grows.
  • Local file persistence is simple, but harder to deploy in production-style hosting.
  • SQLite mirroring is useful, but not yet the read source of truth.

These limitations are not failures. They are part of the learning journey and they help explain the roadmap.

3. Full Rebuild Path From Zero

If a student wanted to rebuild the whole application from nothing, the safest lesson sequence would look like this:

  1. Create a new folder and initialize Git.
  2. Create index.html, styles.css, and app.js.
  3. Build a static dashboard shell with a sidebar, hero, stats, form, and task list section.
  4. Add CSS variables, spacing, card surfaces, buttons, and a dark theme baseline.
  5. Add a simple frontend task array and render cards from it.
  6. Build the form submission flow in frontend JavaScript.
  7. Create a Node server that serves the static frontend.
  8. Add a simple /api/todos endpoint.
  9. Write and read tasks from a CSV file.
  10. Replace fake frontend data with backend fetch calls.
  11. Add update and delete API endpoints.
  12. Add sorting and filters.
  13. Add due-state logic, date validation, and date masks.
  14. Add grouped summaries and task counts.
  15. Add light mode and palette selection.
  16. Add compact view and density mode.
  17. Add SQLite synchronization.
  18. Add import/export.
  19. Add automated tests.
  20. Add OpenAPI and Swagger UI.
  21. Refactor the backend into routes, services, and utils.

4. Suggested Teaching Plan For A Beginner

If you were teaching this to someone from zero, do not start with advanced topics first. Teach in layers:

  1. Web page basics: HTML structure and CSS styling.
  2. Interactivity: JavaScript events and rendering.
  3. Persistence: file-based storage.
  4. APIs: backend routes and JSON.
  5. Quality: tests and documentation.
  6. Architecture: modularization and future database planning.

This order makes the project feel achievable instead of overwhelming.

5. Future Roadmap Context

Current future directions in the project include:

  • frontend smoke tests
  • auto-refresh behavior
  • optional SQLite-primary reads
  • optional Oracle Database 26ai on-premise integration
  • optional Oracle OCI Autonomous Database integration

Notice how these are not random features. They are strategic next steps based on the current architecture.

6. How To Study The Codebase Effectively

  1. Read a lesson here first.
  2. Open the related project file.
  3. Trace one flow from beginning to end.
  4. Write your own small experiments or comments in a separate branch.
  5. Run the app and tests after each change.

Most important advice: do not memorize the code blindly. Instead, learn the purpose of each layer, then the role of each file, then the details of each function. That is how you become able to recreate a project instead of only reading one.
