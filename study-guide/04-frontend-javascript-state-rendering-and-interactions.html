Lesson 4

Frontend JavaScript: State, Rendering, and Interactions

This lesson is about how the interface behaves. It explains the state object, DOM references, event binding, data loading, rendering strategies, localStorage persistence, chart generation, and interactive flows.

Lesson Contents

  • State object
  • DOM references
  • Initialization flow
  • Rendering strategy
  • Filters and sorting
  • Date masking and validation
  • localStorage persistence
  • Chart logic

1. The State Object

The frontend uses a small plain JavaScript object called state to track important UI mode and data information.

const state = {
  items: [],
  editingId: null,
  groupBy: 'status',
  sortBy: 'newest',
  viewMode: 'cards',
  densityMode: 'comfortable'
};

This teaches a foundational frontend concept: state is the source of truth for what the UI should show.

Whenever state changes, the code rerenders the parts of the interface that depend on it.

2. DOM References

Early in app.js, the code collects many DOM references with document.querySelector. This is a practical pattern because it prevents repeatedly searching the DOM for the same elements.

Examples include:

  • form inputs
  • filter controls
  • buttons
  • containers for task lists and grouped results
  • chart containers
  • modal controls

This section is very educational for beginners because it shows the bridge between HTML and JavaScript: every dynamic part of the page needs a reference.

3. Initialization Flow

The frontend startup sequence is carefully ordered. On page load, it restores saved preferences first, then binds events, then applies saved theme settings, then loads data.

The order matters:

  1. restore preferences so the UI knows the right state
  2. bind events so controls are interactive
  3. apply theme and palette so the UI looks correct immediately
  4. load todos and grouped data from the backend

This teaches a general engineering principle: initialization order can affect user experience.

4. Rendering Strategy

The app does not use a frontend framework. Instead, it uses direct rendering functions. Key render functions include:

  • renderTodos()
  • renderSummary()
  • renderGroupResults()
  • renderActiveFilters()
  • renderResultsStrip()
  • renderWorkloadChart()

This teaches that frameworks are useful, but not strictly required for building structured, interactive interfaces.

5. Filters, Sorting, and View Mode

The frontend supports a rich set of UI states:

  • search text
  • status filter
  • priority filter
  • category filter
  • due-state quick filters
  • due-date range
  • sort mode
  • cards versus compact view
  • comfortable versus dense display

The most important learning idea here is composability: a user is not limited to one filter. Multiple filters can be active together, and the UI represents that clearly with active pills.

6. Date Masking and Validation

Date entry is handled with a custom strategy:

  • the visible input is a text field with a DD/MM/YYYY mask
  • a hidden native date picker exists behind it
  • JavaScript formats digits as the user types
  • validation detects impossible dates
  • values are converted to ISO when sending to the backend

This is a very good teaching example because it covers:

  • input event handling
  • blur normalization
  • error display
  • data transformation between display format and storage format

7. localStorage Persistence

The app saves many user preferences into localStorage so the dashboard feels persistent and personal. Examples include:

  • light/dark theme
  • selected color scheme
  • sort mode
  • filters
  • view mode
  • density mode
  • collapsed/expanded panel states

This is a simple but powerful UX improvement because it makes the app feel less temporary.

8. Chart Logic

The workload chart is data-driven. It analyzes visible tasks, groups due dates into a rolling window, and then renders a smooth SVG line plus area fill. The visual dots are placed with an overlay layer so they keep a constant visible size.

That small technical detail teaches something useful: sometimes the visually correct solution is not the most obvious one. SVG circles scaled poorly during browser resizing, so the implementation moved the markers to HTML/CSS overlay elements.

Key lesson: frontend work often involves iteration. A first implementation may be functional, but testing in a real browser reveals what should be improved.
