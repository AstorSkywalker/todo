Chart Masterclass

This lesson teaches the workload chart as both engineering and design: how visible tasks become day buckets, how those buckets become SVG geometry, and how geometry becomes a modern glowing chart.

1. What The Chart Is Actually Showing

The workload chart is not a random decoration. It answers a real question:

How many visible tasks are due on each of the next 14 days?

That matters because the chart must always reflect the currently visible task list, not every task in storage. This is why filters and chart rendering are connected.

2. Bucket Generation

function buildWorkloadBuckets(items) {
  const today = startOfDay(new Date());
  const buckets = Array.from({ length: 14 }, (_, index) => {
    const date = new Date(today);
    date.setDate(today.getDate() + index);
    return {
      key: date.toISOString().slice(0, 10),
      label: ...,
      shortLabel: ...,
      total: 0
    };
  });
}

This function creates 14 day-buckets before it looks at any tasks. Then it walks through the visible items and increments the matching bucket.

The chart therefore begins as a data-shaping problem, not a drawing problem.

3. Turning Data Into Geometry

const width = 760;
const height = 145;
const paddingX = 26;
const paddingTop = 18;
const paddingBottom = 24;

These values define the internal coordinate system. The chart cannot draw intelligently until it knows the drawing area.

const points = buckets.map((bucket, index) => ({
  x: paddingX + (index / (buckets.length - 1)) * chartWidth,
  y: paddingTop + (1 - bucket.total / peak) * chartHeight
}));

This is the key chart formula:

  • x comes from the bucket position in time
  • y comes from the bucket total relative to the peak

In other words, the chart is just data translated into coordinates.

4. Smooth SVG Path Generation

function createSmoothPath(points) {
  if (points.length === 1) {
    return `M ${points[0].x} ${points[0].y}`;
  }

  let path = `M ${points[0].x} ${points[0].y}`;

  for (let index = 0; index < points.length - 1; index += 1) {
    const current = points[index];
    const next = points[index + 1];
    const controlX = (current.x + next.x) / 2;
    path += ` C ${controlX} ${current.y}, ${controlX} ${next.y}, ${next.x} ${next.y}`;
  }

  return path;
}

This function is one of the most beautiful parts of the app because it turns simple points into a smooth curve with cubic Bezier commands.

The learning lesson here is big: advanced visuals often come from a few lines of geometry, not from giant libraries.

6. Responsiveness and Why The Dots Are HTML

Originally the dots were SVG circles, but resizing made them stretch or shrink visually in ways that looked wrong. The fix was to move dots into an HTML overlay.

<div id="workloadChartDots" class="chart-dots-layer"></div>

Why is this better?

  • the SVG can stay responsive
  • the dots can keep a stable pixel size
  • the chart still feels aligned because the dots are positioned from the same computed coordinates

Important chart lesson: sometimes the best visual solution is hybrid. The line can live in SVG while markers live in HTML.

7. Exercise and Checkpoint

Problem

Change the chart from a 14-day window to a 7-day window.

Solution Approach

Update the bucket generator first, because the rest of the chart depends on the number of buckets it receives.

const buckets = Array.from({ length: 7 }, (_, index) => {
  const date = new Date(today);
  date.setDate(today.getDate() + index);
  return {
    key: date.toISOString().slice(0, 10),
    label: ...,
    shortLabel: ...,
    total: 0
  };
});

You do not need to rewrite the geometry formulas because they already adapt to the bucket count.

How It Should Look

The chart should show only seven labels along the bottom, and the line should redistribute itself across the full width of the chart instead of leaving unused space.

Checkpoint: if you can explain how buckets, points, paths, area fill, blur, and overlay dots work together, you really understand how the chart was built.
