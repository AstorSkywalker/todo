Lesson 29

API Design Principles

Learn how the TaskFlow API stays understandable through route naming, validation, result shapes, status codes, and a clear separation between route logic and service logic.

On this page

  • Route design
  • HTTP verbs and intent
  • Response design
  • Validation and errors
  • Why modularity helps API design
  • Common mistakes
  • Exercise and checkpoint

1. Route Design Should Be Clear Before It Is Clever

The current API works well as a learning and portfolio API because the route structure is readable:

GET    /api/todos
POST   /api/todos
PUT    /api/todos/:id
DELETE /api/todos/:id
GET    /api/todos/group
GET    /api/todos/storage
POST   /api/todos/import
GET    /api/todos/export

This is easy to scan because resource, action, and route purpose are explicit.

2. HTTP Verbs Should Match Intent

  • GET retrieves data
  • POST creates or uploads data
  • PUT replaces or updates a known resource
  • DELETE removes a known resource

This consistency helps both humans and tools like Swagger understand the API quickly.

3. Response Design Should Be Predictable

The project uses consistent JSON response patterns. A successful request returns data. A failed request returns an error message and an appropriate status code.

{
  "error": "Title is required."
}

Predictable response shapes matter because they reduce guesswork for frontend code and future API consumers.

4. Validation and Errors Are Part Of Good API Design

A well-designed API is not only about the happy path. It should also reject bad input clearly.

  • missing title
  • invalid status
  • invalid priority
  • invalid date
  • invalid import mode or format

That is why validation helpers matter so much. They help the API behave consistently and transparently.

5. Why Modularity Helps API Design

Routes stay cleaner when they focus on HTTP concerns while services focus on business logic.

route layer
  - read request
  - decide which service to call
  - translate service result into HTTP response

service layer
  - validate
  - filter
  - create/update/delete
  - import/export
  - coordinate storage

This separation makes the API easier to document, test, and extend later.

6. Common Mistakes

  • Making route names too clever: simple and explicit routes are easier to use.
  • Returning inconsistent error shapes: the frontend becomes harder to maintain.
  • Mixing business logic into route files: routes become harder to read and test.
  • Ignoring documentation: an API is stronger when Swagger and OpenAPI describe it clearly.

7. Exercise and Checkpoint

Problem

Describe how you would add a new endpoint to return only overdue tasks without making the API confusing.

Solution Approach

A good answer should reuse existing API patterns instead of inventing a surprising route style.

Option 1:
GET /api/todos?dueState=overdue

Option 2:
GET /api/todos/overdue

Preferred here:
GET /api/todos?dueState=overdue

Why:
- it keeps filtering logic on the list route
- it avoids multiplying small special-case routes
- it matches how filtering already works

How It Should Look

Your answer should feel consistent with the existing API. The strongest design choices usually extend current patterns instead of adding one-off exceptions.

Checkpoint: if you can explain why clear routes, consistent responses, validation, and modular services all matter together, you understand the core API design story of the project.
