Lesson 29

API Design Principles

Learn how the TaskFlow API stays understandable through route naming, validation, result shapes, status codes, and a clear separation between route logic and service logic.

1. Route Design Should Be Clear Before It Is Clever

The current API works well as a learning and portfolio API because the route structure is readable:

GET    /api/todos
POST   /api/todos
PUT    /api/todos/:id
DELETE /api/todos/:id
GET    /api/todos/group
GET    /api/todos/storage
POST   /api/todos/import
GET    /api/todos/export

This is easy to scan because resource, action, and route purpose are explicit.

2. HTTP Verbs Should Match Intent

  • GET retrieves data
  • POST creates or uploads data
  • PUT replaces or updates a known resource
  • DELETE removes a known resource

This consistency helps both humans and tools like Swagger understand the API quickly.

3. Step-by-Step: How To Design One Good Endpoint

The easiest way to learn API design is to walk through one endpoint slowly. Let us use the current list endpoint as the teaching example.

Goal: return a filtered list of todos without making the route system confusing.

  1. Choose the resource: the resource is still todos.
  2. Choose the action style: listing data is a GET request.
  3. Decide where filters belong: filters belong in query parameters, not in random new route names.
  4. Keep the route readable: use /api/todos instead of inventing multiple list-like routes.
  5. Pass only the needed filter values: search, status, priority, and category.
  6. Let the service layer do the real work: route files should not contain all filtering logic.
  7. Return one predictable JSON shape: items plus summary.

The actual route handling already reflects this thinking:

if (req.method === 'GET' && pathParts.length === 2) {
  const result = listTodos({
    search: requestUrl.searchParams.get('search'),
    status: requestUrl.searchParams.get('status'),
    priority: requestUrl.searchParams.get('priority'),
    category: requestUrl.searchParams.get('category')
  });

  sendJson(res, 200, result);
  return;
}

This is a good pattern to study because the route reads almost like plain English:

  • if the request is a list request
  • collect the query filters
  • ask the service layer for the result
  • return the result as JSON

4. Response Design Should Be Predictable

The project uses consistent JSON response patterns. A successful request returns data. A failed request returns an error message and an appropriate status code.

{
  "error": "Title is required."
}

Predictable response shapes matter because they reduce guesswork for frontend code and future API consumers.

For example, the list endpoint returns the same top-level shape every time:

{
  "items": [...],
  "summary": {
    "total": 192,
    "pending": 78,
    "inProgress": 60,
    "done": 54
  }
}

This matters because the frontend can render summary cards and the task list from one stable contract.

5. Validation and Errors Are Part Of Good API Design

A well-designed API is not only about the happy path. It should also reject bad input clearly.

  • missing title
  • invalid status
  • invalid priority
  • invalid date
  • invalid import mode or format

That is why validation helpers matter so much. They help the API behave consistently and transparently.

if (result.error) {
  sendJson(res, result.statusCode, { error: result.error });
  return;
}

This small response pattern is one of the most useful backend habits in the whole project. It keeps error handling short, readable, and uniform.

6. Why Modularity Helps API Design

Routes stay cleaner when they focus on HTTP concerns while services focus on business logic.

route layer
  - read request
  - decide which service to call
  - translate service result into HTTP response

service layer
  - validate
  - filter
  - create/update/delete
  - import/export
  - coordinate storage

This separation makes the API easier to document, test, and extend later.

7. Why Documentation Belongs To Good API Design

A clear API is easier to use when the documentation reflects the actual route design. In this project, the OpenAPI file mirrors the route choices directly.

"/api/todos": {
  "get": {
    "summary": "List tasks",
    "parameters": [
      { "in": "query", "name": "search" },
      { "in": "query", "name": "status" },
      { "in": "query", "name": "priority" },
      { "in": "query", "name": "category" }
    ]
  }
}

This is important because good API design is not only route code. It is also the contract you communicate through Swagger and OpenAPI.

Practical design rule: if an endpoint is hard to describe clearly in OpenAPI, that can be a sign the endpoint itself needs simplification.

8. Common Mistakes

  • Making route names too clever: simple and explicit routes are easier to use.
  • Returning inconsistent error shapes: the frontend becomes harder to maintain.
  • Mixing business logic into route files: routes become harder to read and test.
  • Ignoring documentation: an API is stronger when Swagger and OpenAPI describe it clearly.
  • Creating a new route when a filter would do: not every variation needs a brand-new endpoint.

9. Exercise and Checkpoint

Problem

Describe how you would add a new endpoint to return only overdue tasks without making the API confusing.

Solution Approach

A good answer should reuse existing API patterns instead of inventing a surprising route style.

Option 1:
GET /api/todos?dueState=overdue

Option 2:
GET /api/todos/overdue

Preferred here:
GET /api/todos?dueState=overdue

Why:
- it keeps filtering logic on the list route
- it avoids multiplying small special-case routes
- it matches how filtering already works

How It Should Look

Your answer should feel consistent with the existing API. The strongest design choices usually extend current patterns instead of adding one-off exceptions.

Checkpoint: if you can explain why clear routes, consistent responses, validation, and modular services all matter together, you understand the core API design story of the project.
