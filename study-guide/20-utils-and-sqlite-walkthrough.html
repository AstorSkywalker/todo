1. Reading utils/http.js

function sendJson(res, statusCode, payload) {
  res.writeHead(statusCode, {
    'Content-Type': 'application/json; charset=utf-8',
    'Access-Control-Allow-Origin': '*',
    'Access-Control-Allow-Methods': 'GET,POST,PUT,DELETE,OPTIONS',
    'Access-Control-Allow-Headers': 'Content-Type'
  });
  res.end(JSON.stringify(payload));
}

This helper is tiny, but it is powerful. Every route can now send JSON consistently without rewriting headers again and again.

3. Reading sqliteService.js

let db;
let lastSyncAt = null;

These variables show that the SQLite connection is maintained at module scope and reused. That keeps the service simple for a small project.

db = new Database(DB_FILE);
db.pragma('journal_mode = WAL');

This creates the SQLite database and enables a safer journal mode. Even if you do not know SQLite deeply yet, this is a good example of local database initialization.

function syncTodosToSqlite(todos) {
  const database = initializeSqlite();
  const replaceAll = database.transaction((items) => {
    database.prepare('DELETE FROM todos').run();
    ...
  });
  replaceAll(todos);
  lastSyncAt = new Date().toISOString();
}

This service chooses a replace-all sync strategy. That is a deliberate tradeoff: not the most advanced possible method, but simple, readable, and reliable for the app's scale.

These files are a good reminder that architecture often improves not because files become bigger, but because small files start carrying focused responsibilities.

5. Common Mistakes

  • Underestimating helper files: small shared files often carry a lot of consistency value.
  • Putting validation inside every route: duplicated rules become harder to maintain and test.
  • Making the SQLite mirror too magical: it should stay observable and understandable.
  • Ignoring synchronization visibility: status routes and timestamps make storage behavior easier to debug.

6. Exercise and Checkpoint

Problem

Add one more field to the SQLite summary response, such as a static mode string that says mirror.

Solution Approach

The clean place to do this is in getSqliteSummary(), because that function already defines the shape of the SQLite status response.

function getSqliteSummary() {
  const database = initializeSqlite();
  const total = database.prepare('SELECT COUNT(*) AS total FROM todos').get().total;
  return {
    dbFile: DB_FILE,
    total,
    lastSyncAt,
    mode: 'mirror'
  };
}

How It Should Look

The storage status response should now include a new field inside the SQLite object. This teaches you to extend behavior where responsibility already lives instead of patching it from the outside.

Checkpoint: if you can explain why helpers and storage services deserve their own files, you understand a big part of backend maintainability.
