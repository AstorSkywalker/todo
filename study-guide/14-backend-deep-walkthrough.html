1. Startup Sequence

When the Node server starts, several things happen before the browser ever sees the dashboard. A beginner often thinks a server only listens on a port, but a real application usually has startup work to do first.

ensureCsvFile()
initializeSqlite()
syncCsvToSqlite()
createServer(...)
listen(port)

That startup sequence matters because the dashboard assumes the storage files are already present and consistent. If startup skipped those checks, the app could fail later when the first request arrives.

2. What server.js Really Does

The backend entry point should not do everything. In this project, server.js mostly acts like an orchestrator.

const server = http.createServer(async (req, res) => {
  if (req.url.startsWith('/api/todos')) {
    await handleTodosApi(req, res);
    return;
  }

  await serveStaticAsset(req, res);
});

This is a healthy pattern because the server entry file becomes easy to scan:

  • startup concerns live here
  • API dispatch starts here
  • static file serving starts here
  • business rules do not live here

3. Trace One Route End To End

Let us trace POST /api/todos in slow motion.

  1. The browser serializes the form into JSON.
  2. The request reaches server.js.
  3. server.js delegates to handleTodosApi in routes/todos.js.
  4. The route helper reads the body with readBody(req).
  5. The route calls the service layer.
  6. The service layer normalizes and validates the payload.
  7. The service reads current rows from the CSV service.
  8. The service mutates the array in memory.
  9. The CSV service writes the updated rows back to disk.
  10. The SQLite service mirrors the CSV state.
  11. The route sends the created record as JSON.

Learning technique: if a backend feels confusing, do not stare at all the files at once. Pick one route and follow it through the stack from browser to disk and back.

4. Why The Service Layer Matters

The service layer gives the backend a place for application rules. It is where the backend decides what should happen, not just which URL was requested.

function createTodo(payload) {
  const normalized = normalizeTodoInput(payload);
  validateTodo(normalized);
  const todos = readTodosFromCsv();
  const nextTodo = { ...normalized, id: createId(), ...timestamps };
  todos.push(nextTodo);
  writeTodosToCsv(todos);
  syncTodosToSqlite(todos);
  return nextTodo;
}

Notice how this function reads like a story. That is one of the strongest benefits of modularization: the business rule becomes narratable.

5. CSV and SQLite Together

This project is educational because it uses two storage layers at once:

  • CSV is still the primary source of truth
  • SQLite mirrors the same rows
CSV file
  -> easy to inspect
  -> easy to export
  -> easy for learning

SQLite file
  -> structured relational storage
  -> future migration path
  -> stronger production direction

The point here is not only technical. It also teaches architecture evolution. The app did not need to jump from zero straight to a heavier database. It gained storage sophistication gradually.

6. Validation Pipeline

Validation works best when it is centralized. In this project, the validation helpers keep important rules together.

validateTodo(payload)
isValidIsoDate(value)
normalizeTodoInput(payload)
normalizeImportedTodo(row)

Why is that good?

  • rules stay consistent across create, update, and import
  • tests can focus on one validation module
  • future database migrations reuse the same business rules

7. Why This Architecture Is Healthy

It is healthy because responsibilities are visible:

LayerResponsibilityExample file
BootstrapStart the server, initialize storage, serve filesserver.js
RoutesMatch HTTP requests to backend actionsroutes/todos.js
ServicesRun business logic and coordinate storageservices/todoService.js
StorageRead and write concrete files and database rowsservices/csvService.js, sqliteService.js
HelpersShared low-level logic such as validation and JSON responsesutils/validation.js, utils/http.js

Once responsibilities are this visible, debugging becomes less intimidating. You no longer ask, "Where is everything?" You ask, "Which layer should own this change?"

8. Exercise and Checkpoint

Problem

Add a tiny backend endpoint called GET /api/health that confirms the server is alive.

Solution Approach

Because this endpoint is small and independent from todo logic, you can place it directly in server.js before static file serving.

if (req.url === '/api/health' && req.method === 'GET') {
  sendJson(res, 200, {
    ok: true,
    service: 'taskflow',
    storage: 'csv-primary-sqlite-mirror'
  });
  return;
}

If needed, import sendJson from utils/http.js.

How It Should Look

Opening /api/health in the browser or calling it with curl should return:

{
  "ok": true,
  "service": "taskflow",
  "storage": "csv-primary-sqlite-mirror"
}

Checkpoint: if you can explain why routing, services, storage, and validation are separate, you understand the architectural foundation of the backend.
