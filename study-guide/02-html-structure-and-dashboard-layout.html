Lesson 2

HTML Structure and Dashboard Layout

This lesson teaches how the interface is organized using semantic HTML, what each area of the page does, and why the dashboard is broken into sections instead of one giant block.

Lesson Contents

  • Semantic HTML
  • App shell
  • Sidebar
  • Hero and stats
  • Task form
  • Filters and grouping
  • Task list and templates
  • Docs page

1. Why Semantic HTML Matters

Semantic HTML means choosing elements based on meaning, not only appearance. For example:

  • <aside> is used for the sidebar because it contains supporting information.
  • <main> wraps the primary dashboard experience.
  • <section> groups related content blocks.
  • <article> is used for cards that stand as meaningful independent units.
  • <form>, <label>, <input>, and <textarea> are used correctly for accessibility and browser behavior.

This matters because semantic structure improves accessibility, maintainability, and clarity.

2. The App Shell

The outer page shell gives the app its two major columns:

<div class="app-shell">
  <aside class="sidebar glass-card">...</aside>
  <main class="dashboard">...</main>
</div>

This is one of the most important structural decisions in the project. The sidebar contains identity, themes, APIs, and storage status. The main dashboard contains the actual working interface.

3. Sidebar Structure

The sidebar is not only decorative. It is a utility panel. It contains:

  • project branding and short description
  • theme switcher
  • color scheme selector
  • API summary box
  • storage status panel

This creates a useful left rail that keeps global controls separate from the task workflow.

Why this is a good design: users do not need the theme switch or API list mixed into the task form. Separating global controls from task-specific controls reduces clutter.

4. Hero, KPI Cards, and Chart Placement

At the top of the dashboard, the HTML establishes a visual hierarchy:

  1. hero section for the dashboard title and import/export actions
  2. stats grid for counts
  3. analytics section for the workload chart

This structure teaches a classic dashboard principle: summary first, then analysis, then detailed work surfaces.

5. Task Form HTML

The task form is a complete input surface for creating and editing tasks. Important HTML ideas used here include:

  • hidden input for task id during edits
  • labels that wrap their controls for usability
  • grouped fields using div.form-row
  • a visible masked date input plus a hidden native date picker
  • small inline error elements under fields
<label>
  Due date
  <div class="date-input-wrap">
    <input id="dueDate" type="text" placeholder="DD/MM/YYYY" />
    <input id="dueDatePicker" class="date-picker-native" type="date" />
    <button id="dueDateButton" class="date-icon-button" type="button">...</button>
  </div>
  <small id="dueDateError" class="field-error hidden"></small>
</label>

This pattern is one of the more educational pieces of the UI because it shows how to combine native browser controls with a custom visual experience.

6. Filters and Grouping Sections

The right-hand working column is intentionally split into independent panels:

  • Filters: search, selects, due range, quick due-state chips
  • Grouping: status/priority/category summary
  • Tasks: active filters, view toggles, and the rendered task list

These are separate because they represent different mental jobs:

  • finding
  • analyzing
  • reviewing results

That split greatly improves readability compared with placing every control inside one large card.

7. Task List Templates

Instead of building task cards entirely with string concatenation, the project uses HTML <template> elements. This is a powerful native browser feature.

There are two task templates:

  • a card template for the default rich view
  • a compact row template for the table-like view

Templates are useful because they keep structure visible in HTML while letting JavaScript clone and populate them later.

8. The Documentation Page HTML

The project also includes a separate documentation page, docs.html. This page is not part of the task dashboard itself. It exists to render Swagger UI and now also includes its own theme and palette controls that share settings with the main app.

This teaches a very practical pattern: you can have more than one HTML entry point in the same project, as long as the server knows how to serve them.
