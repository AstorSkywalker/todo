Lesson 21

Deployment and Publishing

This lesson explains how to think about publishing the project in its current state, what works well for portfolio sharing, what gets tricky with file-based persistence, and how the deployment story changes as the storage model evolves.

Lesson Contents

  • GitHub as publication
  • Local demo publishing
  • Live demo hosting
  • Why file persistence changes deployment
  • Future production path
  • Exercise and checkpoint

1. GitHub Is Already A Form Of Publishing

Publishing does not always mean deploying a live URL. For a project like this, GitHub is already part of the publishing story because it provides:

  • source code access
  • screenshots
  • README documentation
  • Swagger/OpenAPI references
  • the study guide

That means the project is already portfolio-ready even before a public live deployment exists.

2. Publishing As A Local Demo

The simplest reliable way to demonstrate the app is still local execution:

npm install
npm start

Then open:

  • http://localhost:3000
  • http://localhost:3000/docs
  • http://localhost:3000/study-guide

This is the most predictable deployment mode because the CSV file and SQLite file are fully under your control on your own machine.

3. Live Demo Hosting

If you want a public URL, the app needs a Node-friendly host because it is not a purely static site. Hosts that can run a Node server are much better candidates than static-only platforms.

4. Why File Persistence Changes Deployment

The app currently stores its primary data in:

  • data/todos.csv
  • data/todos.db as a mirror

That is excellent for local learning, but it changes the hosting story because many cloud platforms treat local disk as temporary or resettable.

Main deployment lesson: an app can be fully valid as a local full-stack app even if its current storage strategy is not yet ideal for production hosting.

This is not a failure of the architecture. It is simply a natural consequence of using file-based persistence.

5. Future Production Path

The future production story becomes stronger as the storage system evolves:

  1. Current state: CSV primary, SQLite mirror
  2. Intermediate future: optional SQLite-primary reads
  3. Advanced future: Oracle Database 26ai on-prem or OCI Autonomous Database
  4. Production-ready hosting: Node app + real database + environment-based configuration

This is a healthy engineering roadmap because it does not force production complexity too early, but it leaves a clear path toward it.

6. Exercise and Checkpoint

Problem

Write a simple publishing recommendation for the project in its current state using only three steps.

Solution Approach

A practical answer should separate portfolio publishing from live hosting realities.

1. Publish the project cleanly on GitHub with README, screenshots, Swagger docs, and study guide.
2. Run it locally for reliable demonstrations because CSV and SQLite work best there today.
3. Use a Node-friendly host only for a demo deployment, and move toward a real database before treating it as production-ready.

How It Should Look

Your answer should feel realistic and honest. A strong deployment explanation does not pretend the current storage model is perfect for production; it explains the tradeoff clearly and proposes the next step.

Checkpoint: if you can explain the difference between publishing a portfolio project, hosting a live demo, and deploying a production-style system, you understand the deployment story of this app.
