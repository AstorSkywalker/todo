Lesson 1

Foundations and Project Setup

This lesson explains what the project is, why it was built the way it was, what every top-level file is responsible for, and how you would set up the same project from an empty folder.

1. Project Purpose

TaskFlow CSV is not just a To Do list. It is a teaching project that demonstrates how a real-world dashboard can grow from a very simple persistence model into a more layered and professional full-stack application.

The core idea was to keep the storage simple enough to understand from day one, which is why the first data layer was a plain CSV file. That decision made the first version easy to reason about because a task could literally be inspected as a line of text in data/todos.csv.

From there, the app added:

  • a modern frontend dashboard
  • a Node.js backend API
  • SQLite mirroring
  • import/export support
  • sorting, filters, and grouped summaries
  • OpenAPI documentation
  • automated tests

Important mindset: the project was not built by starting with the biggest architecture possible. It was built by making something understandable first, then making it more capable.

Each folder has a purpose:

  • data/: persistent local data files
  • routes/: request routing and endpoint handling
  • services/: business logic and storage operations
  • utils/: small reusable helpers
  • scripts/: development utilities, especially tests
  • study-guide/: learning material kept separate from the app

4. How Startup Works

When you run npm start, Node executes server.js. That file is now deliberately small. Its job is to:

  1. ensure storage files exist
  2. initialize SQLite
  3. sync CSV data into SQLite
  4. create the HTTP server
  5. serve static files like index.html, styles.css, app.js, docs.html, and openapi.json
  6. send API requests to the route handler

This is a clean pattern because startup logic stays separate from endpoint logic.

5. Recommended Build Order From Zero

If you were teaching someone from nothing, the order should be practical:

  1. Create index.html, styles.css, and app.js.
  2. Build a static dashboard shell with no data yet.
  3. Create a tiny server.js that serves static files.
  4. Add a simple JSON endpoint like GET /api/todos.
  5. Add file-based persistence with CSV.
  6. Connect the frontend form and list rendering.
  7. Add sorting and filtering.
  8. Add theming and extra UI polish.
  9. Add SQLite mirroring.
  10. Add import/export, tests, and docs.

Teaching principle: build vertical slices. A vertical slice means one feature working end to end, instead of many disconnected pieces.

6. Why CSV First, Then SQLite

CSV was chosen first because it is extremely transparent. A beginner can open the file and inspect exactly what is stored. That gives immediate feedback and reduces fear around persistence.

SQLite was added later because it introduces better structure and a more realistic database story, while still staying local and lightweight.

That storage strategy teaches an important software lesson: you do not need the final architecture on day one. You can evolve your design as the product proves its needs.

7. Environment and Tools

To recreate the project comfortably, you should understand the surrounding tools as well:

  • VS Code: editing, terminal, Markdown preview, source control panel.
  • Git: tracking changes, creating checkpoints, pushing to GitHub.
  • Node.js: running the server and scripts.
  • PowerShell or terminal: running commands like npm start and node scripts/run-tests.js.
  • Browser DevTools: debugging layout, network requests, and localStorage.

In practice, building the app is only part of the skill. Understanding the workflow around it is also part of becoming productive.
