Every write operation begins the same way: normalize, validate, then mutate data. That order matters because it prevents invalid rows from reaching storage.

The creation logic is intentionally simple. For a learning project, a timestamp-based id is enough to keep the code approachable.

This shows the current storage model very clearly: everything is read into memory, changed in memory, and then rewritten to disk.

6. Validation Helpers As A Shared Rulebook

The validation helpers make backend behavior consistent. Even without opening the whole file, you should understand the purpose of each function:

  • normalizePayload() prepares incoming data
  • validateTodo() checks business rules
  • normalizeImportedTodos() cleans file-import rows and resolves ids
  • isValidIsoDate() protects the backend from malformed dates

When rules live together like this, tests become more focused and future storage migrations become easier.

7. Common Mistakes

  • Skipping normalization: raw request data is often inconsistent and harder to validate safely.
  • Mixing validation into many places: shared helpers keep backend rules coherent.
  • Treating CSV writes and SQLite sync as unrelated: in this app they are part of one storage flow.
  • Overcomplicating ids too early: simple ids are fine in a learning project if the tradeoff is understood.
