Lesson 5

Backend Node.js, Routing, Services, and Storage

This lesson explains how the backend works after modularization, why responsibilities were split, and how requests move from the browser into CSV and SQLite persistence.

Lesson Contents

  • Route handling
  • Service layer
  • HTTP helpers
  • Validation helpers
  • Full request flow
  • Why modularization happened

1. The Role of server.js

After the backend refactor, server.js is mostly a bootstrap file. This is exactly what we want. It now focuses on:

  • ensuring the data file exists
  • initializing SQLite
  • synchronizing CSV into SQLite at startup
  • serving static files
  • forwarding API requests into the route module

A small server entry point is healthier than a giant one because it keeps startup concerns separate from business logic.

2. Route Handling

The file routes/todos.js is responsible for deciding what to do when the request path points to /api/todos.

It does not know how CSV parsing works internally. It does not know how grouping is implemented. It only knows which service to call and which HTTP response to send.

This is an example of separation of concerns. Routes are traffic controllers, not storage engines.

3. Service Layer

The service layer is where business logic lives.

services/todoService.js handles:

  • listing tasks
  • getting a task by id
  • creating tasks
  • updating tasks
  • deleting tasks
  • grouping
  • storage status
  • import/export

services/csvService.js handles:

  • ensuring the CSV file exists
  • reading tasks from the CSV file
  • writing tasks to the CSV file
  • CSV line parsing
  • CSV value escaping

4. HTTP Helpers

utils/http.js contains small cross-cutting helpers:

  • readBody(req) to read and parse JSON request bodies
  • sendJson(res, statusCode, payload) to return JSON with consistent headers

Small helpers like these reduce repetition and make route files easier to read.

5. Validation Helpers

utils/validation.js centralizes rules such as:

  • required title
  • allowed status values
  • allowed priority values
  • valid ISO dates
  • payload normalization
  • import normalization

This is valuable because validation should not be scattered randomly across the codebase.

6. Full Request Flow Example

Take POST /api/todos as an example:

  1. The browser sends JSON to the server.
  2. server.js routes the request to handleTodosApi.
  3. routes/todos.js reads the body and calls createTodo.
  4. services/todoService.js normalizes and validates the payload.
  5. The service reads current tasks from services/csvService.js.
  6. The new task is inserted into the array.
  7. The updated task array is written back to CSV.
  8. CSV writing triggers SQLite synchronization.
  9. The service returns the created task.
  10. The route sends it back as JSON.

This is the kind of end-to-end flow a student should visualize for every endpoint.

7. Why Modularization Happened Later

At the beginning, a single server.js was good enough. That is not a mistake. It was appropriate for the project size at the time.

Modularization happened after the backend proved it had enough complexity to justify it. That timing is important. Over-engineering too early can make learning harder, not easier.

Healthy refactor rule: split code when responsibilities become blurry, testing becomes harder, or file size starts harming readability. That is exactly what happened here.
